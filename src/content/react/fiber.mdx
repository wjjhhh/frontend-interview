## 为什么需要 Fiber？

React 15 使用递归的 **Stack Reconciler**，一旦开始 diff 就无法中断。当组件树很深时，主线程长时间被占用，导致：
- 动画掉帧（帧率 < 60fps）
- 用户输入响应延迟
- 页面感觉"卡顿"

Fiber 的目标：将渲染工作拆成**可中断的小单元**，让浏览器有机会处理高优先级任务。

## Fiber 节点结构

每个 React 元素对应一个 Fiber 节点，通过链表连接整棵树：

```javascript
{
  // 节点身份
  tag: WorkTag,           // 组件类型（函数组件/类组件/Host等）
  key: null | string,
  type: any,              // 组件函数 或 DOM 标签字符串
  stateNode: any,         // 真实 DOM 节点 或 类组件实例

  // 树结构（三叉链表，非递归遍历）
  return: Fiber | null,   // 父节点
  child: Fiber | null,    // 第一个子节点
  sibling: Fiber | null,  // 下一个兄弟节点

  // 双缓冲
  alternate: Fiber | null, // 指向另一棵树的对应节点

  // 状态与副作用
  memoizedState: any,     // Hooks 链表 / 类组件 state
  memoizedProps: any,
  pendingProps: any,
  flags: Flags,           // 副作用标记（Placement/Update/Deletion）
  subtreeFlags: Flags,

  // 调度
  lanes: Lanes,           // 本节点的优先级
  childLanes: Lanes,      // 子树优先级
}
```

## 高频面试题

### 1. Fiber 的遍历算法

Fiber 用**深度优先 + 链表**替代递归调用栈，遍历可随时中断：

```text
遍历顺序（performUnitOfWork）：

       A
      / \
     B   C
    / \
   D   E

1. beginWork(A) → 向下
2. beginWork(B) → 向下
3. beginWork(D) → 无子节点
4. completeWork(D) → 找 sibling
5. beginWork(E) → 无子节点
6. completeWork(E) → 无 sibling，回到 return
7. completeWork(B) → 找 sibling
8. beginWork(C) → 无子节点
9. completeWork(C) → 无 sibling，回到 return
10. completeWork(A) → 完成
```

每处理完一个节点检查：**是否还有剩余时间**（`shouldYield()`）。没有则暂停，等下一帧继续。

**为什么是深度优先，而非广度优先（BFS）？**

React 的工作单元是**子树**而非**层级**，DFS 有三个不可替代的优势：

1. **subtreeFlags 自然冒泡**：`completeWork(B)` 时 B 的整个子树已完成，可以一行代码把子节点的副作用标记汇总到父节点：
   ```javascript
   parent.subtreeFlags |= child.subtreeFlags | child.flags;
   ```
   BFS 处理父节点时子节点还未完成，无法做这个汇总。

2. **Effect 执行顺序正确**：`useLayoutEffect` / `componentDidMount` 必须**子先父后**触发。DFS 的 completeWork 阶段天然是后序遍历（叶→根），完美匹配；BFS 做不到。

3. **内存占用低**：DFS 只需通过 `return` 指针回溯当前路径，O(树深度)；BFS 需要队列存储每层所有节点，O(树宽度)——组件树的宽度通常远大于深度。

**追问：为什么 completeWork 必须先处理完子树，再处理父节点？**

根本原因是 completeWork 要**构建真实 DOM 树**，这是天然自底向上的操作：

```text
<div>           ← 第3步：创建 div，把 span appendChild 进来
  <span>        ← 第2步：创建 span，把文本节点 appendChild 进来
    hello       ← 第1步：创建文本节点
  </span>
</div>
```

父节点的 `appendChild(child)` 必须在子节点已创建后才能调用。如果先 completeWork(div)，子节点还不存在，div 就是空壳。

三件事都强制要求子树先完成：

| 操作 | 原因 |
|------|------|
| **构建 DOM 树** | `parent.appendChild(child)` 要求 child 已存在 |
| **subtreeFlags 汇总** | 父节点要读取子树的副作用标记，子树未完成则数据缺失 |
| **commit 阶段剪枝** | `if (root.subtreeFlags === 0) return` 跳过整棵树；subtreeFlags 必须正确冒泡才能生效 |

### 2. 双缓冲机制（Double Buffering）

React 同时维护**两棵 Fiber 树**：

- **current 树**：当前渲染到屏幕上的内容
- **workInProgress 树**：后台正在构建的新树

```text
首次渲染：
  null ← fiberRoot.current
  构建 workInProgress 树 → commit → current 指向它

更新时：
  current 树 ← fiberRoot.current
  以 current 为基础，创建 workInProgress 树（复用节点）
  commit 后，两树互换角色
```

好处：构建过程不影响当前显示；出错可回退；节点内存复用。

### 3. Lane 优先级模型

React 18 用 **Lane**（二进制位掩码）表示优先级，支持批量处理同优先级任务：

```javascript
// 优先级从高到低（部分）
const SyncLane            = 0b0000000000000000000000000000001; // 同步（不可中断）
const InputContinuousLane = 0b0000000000000000000000000000100; // 连续输入
const DefaultLane         = 0b0000000000000000000000000010000; // 普通更新
const TransitionLane1     = 0b0000000000000000000000001000000; // startTransition
const IdleLane            = 0b0100000000000000000000000000000; // 空闲
```

位操作让优先级合并/比较极其高效：

```javascript
// 判断是否包含某优先级
const includesSomeLane = (a, b) => (a & b) !== 0;

// 合并优先级
const mergeLanes = (a, b) => a | b;

// 移除优先级
const removeLanes = (set, subset) => set & ~subset;
```

### 4. Reconciliation（Diff 算法）核心规则

React 的 Diff 基于三个假设：

1. **同层比较**：不跨层级移动节点
2. **类型不同直接替换**：`<div>` → `<span>` 不复用
3. **key 标识稳定性**：列表必须有稳定的 key

```javascript
// key 的重要性
// ❌ 用 index 做 key，会导致错误复用
{items.map((item, index) => <Item key={index} data={item} />)}

// ✅ 用唯一 ID
{items.map((item) => <Item key={item.id} data={item} />)}
```

## 并发特性

### startTransition：区分紧急/非紧急更新

```javascript
function SearchPage() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);
  const [isPending, startTransition] = useTransition();

  function handleChange(e) {
    // 紧急：立即更新输入框
    setQuery(e.target.value);

    // 非紧急：可被打断的搜索结果更新
    startTransition(() => {
      setResults(heavySearch(e.target.value));
    });
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      {isPending ? <Spinner /> : <ResultList results={results} />}
    </>
  );
}
```

## 深入追问

### React 调度器（Scheduler）如何实现时间切片？

React 没有用 `requestIdleCallback`（兼容性差、最小 50ms 延迟），而是用 **`MessageChannel`** 模拟：

```javascript
const channel = new MessageChannel();
const port = channel.port2;

channel.port1.onmessage = performWorkUntilDeadline;

function scheduleWork() {
  port.postMessage(null); // 触发宏任务，在下一帧开始前执行
}

function performWorkUntilDeadline() {
  const deadline = currentTime + 5; // 每帧最多执行 5ms
  while (workQueue.length && currentTime < deadline) {
    performNextWork();
  }
  if (workQueue.length) scheduleWork(); // 还有任务，下帧继续
}
```

### React 19 的编译器（React Compiler）

React Compiler（原 React Forget）自动插入 `useMemo`/`useCallback`，开发者无需手动优化：

```javascript
// 你写的代码
function Component({ list }) {
  const sorted = list.sort((a, b) => b - a);
  return <List items={sorted} />;
}

// 编译器自动转换（等价于）
function Component({ list }) {
  const sorted = useMemo(() => list.sort((a, b) => b - a), [list]);
  return <List items={sorted} />;
}
```

## 进阶面试题

### 5. 多节点 Diff 算法（列表更新）

单节点 Diff 简单，难点在**多节点**列表的高效更新。React 采用**两轮遍历**：

**第一轮：从左到右，处理顺序未变的常见情况**

```text
old: A → B → C → D
new: A → B → E → D

逐个对比（key + type 均相同才复用）：
- A vs A → 复用 ✅
- B vs B → 复用 ✅
- C vs E → key 不同 → 停止第一轮
```

**第二轮：处理位置变动的节点**

```text
将剩余 oldFibers (C, D) 放入 Map<key, Fiber>
继续遍历剩余 newElements：
- E：Map 中无 key=E → 新建
- D：Map 中有 key=D → 复用，从 Map 删除

遍历结束，Map 中仍剩余 C → 标记删除
```

**lastPlacedIndex 优化（判断节点是否需要移动）**

```text
规则：记录上一个"不需要移动"节点在旧树中的 index (lastPlacedIndex)
      当前节点的 oldIndex < lastPlacedIndex → 需要移动
      否则 → 不移动，更新 lastPlacedIndex

例：old: A(0) B(1) C(2) D(3)  →  new: D A B C
- D: oldIndex=3 >= lastPlacedIndex=0 → 不移动，lastPlacedIndex=3
- A: oldIndex=0 < 3 → 移动
- B: oldIndex=1 < 3 → 移动
- C: oldIndex=2 < 3 → 移动

结论：DOM 操作 = D 不动，ABC 移动到 D 后面（3次 DOM 操作）
```

> ⚠️ 这就是为什么"将末尾元素移到开头"代价远高于"将开头元素移到末尾"——前者导致其余所有节点都需要移动。

### 6. StrictMode 为何双重调用 Effects？

```tsx
// StrictMode 在开发环境故意执行两次：
// mount → unmount → mount（模拟组件快速挂载/卸载）
// 目的：暴露没有正确清理副作用的问题

// ❌ StrictMode 下会创建两个 WebSocket 连接
useEffect(() => {
  const ws = new WebSocket('wss://...');
  ws.onmessage = handleMessage;
  // 缺少清理函数 → 第二次 mount 时旧连接未关闭
}, []);

// ✅ 始终返回清理函数，StrictMode 验证清理是否正确
useEffect(() => {
  const ws = new WebSocket('wss://...');
  ws.onmessage = handleMessage;
  return () => ws.close();
}, []);
```

双重调用**只在开发模式**发生，生产环境不受影响。如果在 StrictMode 下行为异常，说明副作用逻辑有缺陷，需要修复。

受影响的 API：
- `useEffect` / `useLayoutEffect`（会执行 setup → cleanup → setup）
- 函数组件 render 函数（会调用两次，确保幂等）
- `useState` / `useMemo` 的初始化函数

### 7. Error Boundary 工作原理

```tsx
// 函数组件无法成为 Error Boundary，必须用类组件
class ErrorBoundary extends React.Component<
  { fallback: React.ReactNode; children: React.ReactNode },
  { hasError: boolean; error: Error | null }
> {
  state = { hasError: false, error: null };

  // 渲染阶段捕获错误，返回新 state（纯函数，不能有副作用）
  static getDerivedStateFromError(error: Error) {
    return { hasError: true, error };
  }

  // commit 阶段，可以记录日志（可以有副作用）
  componentDidCatch(error: Error, info: React.ErrorInfo) {
    console.error('ErrorBoundary caught:', error, info.componentStack);
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback;
    }
    return this.props.children;
  }
}
```

**Error Boundary 能捕获 vs 不能捕获的错误：**

| 场景 | 是否捕获 |
|------|---------|
| 子组件渲染（render/函数体） | ✅ |
| `useLayoutEffect` / `componentDidMount` | ✅ |
| 事件处理函数 | ❌（需自行 try/catch） |
| 异步代码（setTimeout、Promise） | ❌ |
| Error Boundary 自身 | ❌ |
| SSR | ❌ |

```tsx
// 用高阶函数封装，方便使用
function withErrorBoundary<P extends object>(
  Component: React.ComponentType<P>,
  fallback: React.ReactNode
) {
  return function WrappedComponent(props: P) {
    return (
      <ErrorBoundary fallback={fallback}>
        <Component {...props} />
      </ErrorBoundary>
    );
  };
}

// 或使用 react-error-boundary 库（支持 resetKeys、onReset 等）
import { ErrorBoundary } from 'react-error-boundary';

function App() {
  return (
    <ErrorBoundary
      fallbackRender={({ error, resetErrorBoundary }) => (
        <div>
          <p>出错了：{error.message}</p>
          <button onClick={resetErrorBoundary}>重试</button>
        </div>
      )}
      onError={(error, info) => logError(error, info)}
    >
      <ChildComponent />
    </ErrorBoundary>
  );
}
```

## React Fiber vs Vue 3 渲染机制对比

这是面试中最常见的跨框架横向对比题，核心差异体现在**更新策略**、**Diff 算法**和**编译优化**三个维度。

### 1. 更新触发方式：Pull vs Push

| | React Fiber | Vue 3 |
|---|---|---|
| **模型** | Pull（拉取） | Push（推送） |
| **如何知道变化** | 不知道——整个组件子树重新执行 | 精确知道——响应式系统追踪依赖 |
| **优化手段** | `memo` / `useMemo` / `shouldComponentUpdate` | 自动，无需手动优化 |

```text
React 更新流程：
  setState() → 标记整个组件子树为 dirty → re-render 子树 → diff

Vue 3 更新流程：
  state.value = x → Proxy setter 触发 → 只通知依赖该值的 effect → 精准 re-render
```

Vue 的 Proxy 响应式系统在**依赖收集阶段**就建立了"数据 → 组件"的映射，更新时直接命中，无需从根节点向下 diff。React 的 `useMemo`/`memo` 本质上是在手动模拟这套追踪机制。

### 2. 调度策略：时间切片 vs 微任务队列

**React**：用 `MessageChannel`（宏任务）实现时间切片，每 5ms 让出主线程：

```text
帧 1: [渲染 5ms] [浏览器绘制] [渲染 5ms] ...
       └─ Fiber 可中断  ─┘
```

**Vue 3**：用 `Promise.then`（微任务）批量刷新，同一个事件循环内的所有更新**合并后同步执行**：

```javascript
// Vue 的 nextTick 内部实现（简化）
const queue = [];
let isFlushing = false;

function queueFlush() {
  if (!isFlushing) {
    isFlushing = true;
    Promise.resolve().then(flushJobs); // 微任务，当前 tick 结束后立即执行
  }
}

function flushJobs() {
  // 一次性执行所有 queued 的组件更新
  queue.sort((a, b) => a.id - b.id); // 父组件先于子组件更新
  for (const job of queue) job();
  isFlushing = false;
}
```

| 维度 | React 时间切片 | Vue 微任务队列 |
|------|--------------|--------------|
| **可中断** | ✅（可暂停恢复） | ❌（当帧内同步完成） |
| **延迟** | 稍高（宏任务） | 极低（微任务） |
| **适用场景** | 超复杂 UI、动画不掉帧 | 大多数业务场景 |
| **实现复杂度** | 高（Fiber + Scheduler） | 低 |

> Vue 认为：响应式系统的精准更新让时间切片在大多数场景下变得不必要——需要更新的组件本来就很少。

### 3. Diff 算法：单端 + Map vs 双端 + 最长递增子序列

**React（单端 + Map）**：

```text
old: A → B → C → D → E
new: A → C → B → E → F

第一轮从左扫：A匹配→继续，C≠B→停止
第二轮：剩余 old(B,C,D,E) 建 Map，遍历 new(C,B,E,F)：
  C → Map命中(oldIdx=2) → 复用
  B → Map命中(oldIdx=1) → 需要移动（1 < lastPlacedIndex=2）
  E → Map命中(oldIdx=4) → 复用
  F → Map未命中 → 新建
  D → 未被消费 → 删除
```

**Vue 3（双端对比 + 最长递增子序列）**：

```text
old: A → B → C → D → E
new: A → C → B → E → F

Step 1: 头头对比 → A=A，移动头指针
Step 2: 尾尾对比 → E≠F，停止
Step 3: 中间乱序部分 (B,C,D) vs (C,B,E,F)：
  建立 new 的 key→index 映射
  遍历 old 中间节点，记录 source 数组：[1, 0, -1]（-1 代表新建）
  对 source 求最长递增子序列（LIS）= [0,1] → 对应 C,B 不需要移动
  只移动不在 LIS 中的节点
```

```javascript
// Vue 3 核心：求最长递增子序列（贪心+二分，O(n log n)）
function getSequence(arr) {
  const result = [0];
  for (let i = 1; i < arr.length; i++) {
    if (arr[i] > arr[result[result.length - 1]]) {
      result.push(i);
    } else {
      // 二分查找替换位置
      let lo = 0, hi = result.length - 1;
      while (lo < hi) {
        const mid = (lo + hi) >> 1;
        arr[result[mid]] < arr[i] ? (lo = mid + 1) : (hi = mid);
      }
      result[lo] = i;
    }
  }
  return result; // 不需要移动的节点下标集合
}
```

| 维度 | React Diff | Vue 3 Diff |
|------|-----------|-----------|
| **算法** | 单端扫描 + HashMap | 双端 + LIS |
| **最小移动** | 不保证最少 DOM 操作 | 通过 LIS 保证最少移动次数 |
| **时间复杂度** | O(n) | O(n log n)（LIS 部分） |
| **适用** | 节点变化大、顺序乱 | 节点变化小、顺序部分保留 |

### 4. 编译器优化：运行时 vs 编译时

Vue 3 的模板编译器可以在**编译阶段**静态分析出哪些节点会变、哪些不会，生成带 `patchFlag` 的 VNode：

```javascript
// 你写的模板
// <div class="container">
//   <h1>标题</h1>            <!-- 永远不变 -->
//   <p>{{ msg }}</p>         <!-- 只有文本会变 -->
//   <input :value="val" />   <!-- 只有 value 属性会变 -->
// </div>

// 编译产物（简化）
createBlock('div', { class: 'container' }, [
  createVNode('h1', null, '标题'),           // 无 patchFlag → 跳过 diff
  createVNode('p', null, msg, 1 /* TEXT */), // patchFlag=1 → 只 diff 文本
  createVNode('input', { value: val }, null, 8 /* PROPS */, ['value']),
])
```

**Block Tree（块树）**：`v-if`/`v-for` 会创建新 Block，动态节点被收集到 Block 的 `dynamicChildren` 中，diff 时**只遍历动态子节点**，跳过静态结构：

```text
传统 VNode diff：遍历整棵树（O(n)）
Vue 3 Block diff：只遍历 dynamicChildren（接近 O(动态节点数)）
```

React 没有模板编译器（JSX 是语法糖，不做静态分析），React Compiler（React 19）才开始补齐这一差距，但仍是运行时插入 memo，不如 Vue 的编译时分析彻底。

### 5. 一句话总结

| 维度 | React Fiber | Vue 3 |
|------|-------------|-------|
| **核心思路** | 可中断的异步渲染 | 精准的响应式追踪 |
| **Diff 策略** | 单端扫描，不保证最少操作 | 双端+LIS，最少 DOM 移动 |
| **性能优化** | 运行时（memo/useMemo） | 编译时（patchFlag/Block） |
| **调度** | 宏任务时间切片 | 微任务批量刷新 |
| **复杂度** | 高（但支持并发特性） | 低（开箱即得精准更新） |

> **面试结论**：React 用架构复杂度换取了并发渲染能力（适合复杂交互场景）；Vue 3 用响应式精准追踪 + 编译优化换取了更低的运行时开销（大多数业务场景效率更高）。两者没有绝对优劣，选择取决于场景。
