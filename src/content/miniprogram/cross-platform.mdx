## 为什么需要跨端框架？

原生小程序开发有明显的局限：

```text
痛点：
  - 语法与 Web 差异大（WXML/WXSS，非 HTML/CSS）
  - 各平台（微信/支付宝/抖音/百度）API 不统一
  - 不能复用 Web 端组件和生态
  - 同一功能要在多端重复开发

跨端框架的目标：
  Write Once, Run Everywhere
  ─ 用 React 或 Vue 语法写一份代码
  ─ 编译输出到微信、支付宝、H5、App 等多个平台
```

## 高频面试题

### 1. Taro 编译原理

Taro 3.x（当前主流版本）采用**运行时适配**方案，与 Taro 1/2 的纯编译时方案有本质区别。

#### Taro 1/2：AST 编译时转换

```text
React JSX 代码
    │
    ├─ 通过 Babel AST 转换
    │
    ├─ 将 React 组件 → WXML 模板
    ├─ 将 CSS-in-JS / SASS → WXSS
    └─ 将 React 事件 → 小程序事件绑定

问题：
  - React 语法支持不完整（很多动态特性无法编译）
  - JSX 表达式支持受限（不能随意写复杂逻辑）
```

#### Taro 3：运行时适配（主流）

```text
核心思路：在小程序环境中模拟 DOM / BOM API，让 React 能在小程序里运行

架构：
  ┌──────────────────────────────────────────────┐
  │  开发者代码（React / Vue 3 组件）              │
  └──────────────────┬───────────────────────────┘
                     │ 使用标准 React/Vue API
  ┌──────────────────▼───────────────────────────┐
  │  @tarojs/react                               │
  │  模拟 react-dom，将 createElement 调用         │
  │  转为操作 Taro 的虚拟节点树                    │
  └──────────────────┬───────────────────────────┘
                     │
  ┌──────────────────▼───────────────────────────┐
  │  @tarojs/runtime（运行时）                    │
  │  实现 TaroElement / TaroDocument              │
  │  将虚拟 DOM 操作转为 setData 调用              │
  └──────────────────┬───────────────────────────┘
                     │ setData({ root: vNodeTree })
  ┌──────────────────▼───────────────────────────┐
  │  小程序渲染层                                  │
  │  一个超级模板递归渲染 vNodeTree                 │
  └──────────────────────────────────────────────┘
```

```javascript
// 开发者写的 React 代码
function Counter() {
  const [count, setCount] = useState(0);
  return <View onClick={() => setCount(c => c + 1)}>{count}</View>;
}

// 运行时将上述代码中对 DOM 的操作（如 appendChild）
// 转换成对小程序 data 的更新（setData），
// 然后用小程序模板渲染这颗数据树
```

**Taro 3 的优势与代价：**

```text
优势：
  ✅ React 语法完整支持（Hooks、Context 等都能用）
  ✅ 复用 React 生态（react-query、zustand 等）

代价：
  ❌ 运行时包体积增大（小程序主包压力更大）
  ❌ 性能略低于原生（多一层运行时 + setData 转换）
  ❌ 启动时间增加（需初始化运行时）
```

### 2. uni-app 编译原理

uni-app 以 **Vue 语法**为 DSL，也分编译时和运行时两层。

```text
编译阶段（构建时）：
  Vue SFC (.vue)
    │
    ├─ template → 各平台模板（WXML / AXML / swan）
    ├─ script   → 各平台 JS（保留 Vue 响应式核心）
    └─ style    → 各平台样式（WXSS / ACSS）

运行时（小程序中）：
  uni-app 内置了一个精简版 Vue 运行时
  Vue 的响应式（ref/reactive）→ 驱动 setData

跨端兼容层：
  封装各平台 API 为统一接口
  wx.request / my.request / fetch → uni.request
```

```javascript
// uni-app 代码（Vue 3 Composition API）
<script setup>
import { ref } from 'vue';
const count = ref(0);

// 条件编译：针对不同平台写特定代码
// #ifdef MP-WEIXIN
const wxSpecificFn = () => { /* 微信专属逻辑 */ };
// #endif

// #ifdef H5
const h5SpecificFn = () => { /* H5 专属逻辑 */ };
// #endif
</script>

<template>
  <view @click="count++">{{ count }}</view>
</template>
```

**条件编译**是 uni-app 的核心特性，允许在同一份代码中为不同平台写差异逻辑：

```javascript
// JS 中的条件编译
// #ifdef MP-WEIXIN
wx.showToast({ title: '微信专属 API' });
// #endif

// #ifdef H5
alert('H5 环境');
// #endif

// #ifndef MP-WEIXIN  （非微信小程序时执行）
console.log('不是微信小程序');
// #endif
```

### 3. Taro vs uni-app 选型对比

| 对比项 | Taro 3 | uni-app |
|--------|--------|---------|
| 开发语言 | React / Vue 3 | Vue 2 / Vue 3 |
| 维护方 | 京东凹凸实验室 | DCloud |
| 跨端范围 | 小程序、H5、RN | 小程序、H5、App（含原生渲染） |
| 性能 | 中等（运行时开销） | 中等（Vue 运行时） |
| 生态 | React 生态复用 | Vue 生态 + uni 插件市场 |
| 原生 App 支持 | React Native | Weex / nvue（原生渲染） |
| 学习曲线 | 熟悉 React 即可 | 熟悉 Vue 即可 |

**选型建议：**

```text
团队以 React 为主 → Taro
团队以 Vue 为主   → uni-app
需要发布原生 App + 小程序 → uni-app（更成熟的 App 支持）
已有 React 组件库要复用  → Taro
社区插件生态要求高       → uni-app（插件市场更丰富）
```

### 4. 跨端框架的性能瓶颈

#### Taro 3 的 setData 转换开销

```text
普通 React 更新流程：
  state 变化 → React reconcile → DOM diff → 更新真实 DOM

Taro 3 更新流程：
  state 变化 → React reconcile → 操作 TaroDOM（虚拟）
             → 序列化为 JSON → setData → JSBridge
             → 渲染层 diff → 更新 DOM

多了序列化 + JSBridge 两步，数据量大时明显慢于原生
```

#### 优化策略

```javascript
// 1. 避免在循环中触发 setState（会产生多次 setData）
// ❌
items.forEach(item => {
  this.setState(prev => ({ list: [...prev.list, item] }));
});
// ✅ 合并更新
this.setState({ list: [...this.state.list, ...items] });

// 2. 使用 shouldComponentUpdate / React.memo 减少不必要的重渲染
const Item = React.memo(({ data }) => <View>{data.name}</View>);

// 3. 使用 Taro.nextTick 确保 DOM 更新后再操作
this.setState({ show: true }, () => {
  Taro.nextTick(() => {
    // DOM 已更新
  });
});
```

### 5. 原生小程序 vs 跨端框架 选型

```text
选择原生小程序：
  ✅ 性能要求极高（游戏、直播、复杂动画）
  ✅ 只需支持微信小程序一个平台
  ✅ 团队规模小，学习成本优先

选择跨端框架：
  ✅ 需要同时支持多个小程序平台（微信 + 支付宝 + 抖音）
  ✅ 已有 React / Vue Web 端代码要复用
  ✅ 团队更熟悉 Web 开发生态
  ✅ 快速上线，开发效率优先
```

### 6. 小程序与 H5 的互相跳转

```javascript
// 小程序跳转到 H5（WebView）
wx.navigateTo({
  url: '/pages/webview/index?url=' + encodeURIComponent('https://example.com'),
});

// webview 页面
<web-view src="{{url}}" bindmessage="onMessage" />

// H5 页面内跳回小程序（需要在 H5 中引入 jweixin-module）
wx.miniProgram.navigateTo({ url: '/pages/home/index' });
wx.miniProgram.postMessage({ data: { result: 'ok' } }); // H5 → 小程序通信
```

// 从 H5 拉起小程序（微信开放标签，仅在微信 H5 中有效）
