## 什么是微前端？

微前端借鉴了后端微服务的理念，将一个庞大的前端应用拆分为多个**独立开发、独立部署、独立运行**的子应用，再由一个主应用（基座）将它们组合呈现。

```text
传统巨石应用：
  团队 A ─┐
  团队 B ─┼──► 同一个仓库 ──► 一次构建 ──► 一个发布
  团队 C ─┘
  问题：耦合重、发布慢、技术栈无法独立演进

微前端架构：
  团队 A → 子应用 A（React）─┐
  团队 B → 子应用 B（Vue）  ─┼──► 主应用（基座）──► 用户看到完整页面
  团队 C → 子应用 C（Angular）┘
  优势：独立迭代、独立发布、技术栈自由
```

## 高频面试题

### 1. 微前端解决了哪些问题？

| 问题 | 说明 |
|------|------|
| **巨石应用难以维护** | 代码库庞大，构建缓慢，改一处影响全局 |
| **多团队协作冲突** | 多团队共享一个仓库，合并频繁冲突 |
| **技术栈无法升级** | 旧系统用 jQuery/Angular.js，无法整体迁移 |
| **发布相互阻塞** | 一个团队 Bug 导致整个产品无法发布 |

核心价值：**技术无关、独立部署、渐进迁移**。

### 2. 主流实现方案对比

#### 方案一：iframe

```text
原理：每个子应用运行在独立 iframe 中，天然隔离

优点：
  ✅ 隔离最彻底（JS/CSS/DOM 完全隔离）
  ✅ 接入成本最低，子应用无需改造
  ✅ 适合安全要求高的场景（银行、政务）

缺点：
  ❌ URL 同步困难（刷新丢失子路由）
  ❌ 弹窗/Modal 无法突破 iframe 边界
  ❌ 性能差（每次切换重新加载页面）
  ❌ 与主应用通信繁琐（postMessage）
  ❌ SEO 不友好
```

#### 方案二：single-spa / qiankun

```text
原理：统一的生命周期协议，子应用注册到主框架，按路由动态挂载/卸载

优点：
  ✅ 用户体验好（SPA 导航，无白屏）
  ✅ 子应用共享主应用的路由、全局状态
  ✅ qiankun 提供 JS 沙箱和 CSS 隔离

缺点：
  ❌ 子应用需要改造（暴露生命周期钩子）
  ❌ 沙箱不能覆盖所有边界情况
  ❌ 多框架并存时性能有损耗
```

#### 方案三：Webpack Module Federation

```text
原理：构建时将模块联邦化，运行时跨应用共享模块（代码级别复用）

优点：
  ✅ 真正的运行时共享（React/组件库只加载一份）
  ✅ 无需路由级别划分，可以共享任意粒度的模块
  ✅ 构建工具原生支持，对业务代码侵入最小

缺点：
  ❌ 强依赖 Webpack 5（Vite 需要插件）
  ❌ 版本协商复杂，共享依赖版本不匹配难调试
  ❌ 本地开发需要多个服务同时运行
```

#### 方案四：Web Components

```text
原理：子应用封装为自定义 HTML 元素，浏览器原生支持隔离

优点：
  ✅ 标准化，无框架依赖
  ✅ Shadow DOM 提供天然 CSS 隔离

缺点：
  ❌ 与 React/Vue 整合有摩擦
  ❌ 服务端渲染支持差
  ❌ 兼容性（IE 完全不支持）
```

### 3. 如何选型？

```text
场景 1：遗留系统渐进迁移
  推荐：qiankun + 路由划分
  理由：接入成本可控，能保留旧系统正常运行

场景 2：多团队共建大型平台（如中后台）
  推荐：Module Federation
  理由：共享组件库、共享 React，减少重复加载

场景 3：强隔离需求（第三方应用嵌入、安全要求高）
  推荐：iframe
  理由：隔离最彻底，安全性最高

场景 4：全新项目、团队技术栈统一
  推荐：monorepo（Nx/Turborepo）+ 按需拆包
  理由：不一定需要微前端，monorepo 更简单
```

> 面试关键点：**微前端不是银弹**，小团队、技术栈统一的项目用微前端反而增加复杂度。

### 4. 微前端的核心挑战

#### JS 隔离

子应用的全局变量（`window.xxx`）、事件监听、定时器可能互相污染。

```javascript
// 问题：子应用 A 污染全局
window.user = { name: 'A的用户' };

// 子应用 B 读到了错误的值
console.log(window.user); // { name: 'A的用户' } ← 污染！
```

解决方案：JS 沙箱（见 qiankun 篇）

#### CSS 隔离

```css
/* 子应用 A 的样式 */
.btn { background: red; }

/* 子应用 B 同名类名 —— 互相覆盖！ */
.btn { background: blue; }
```

解决方案：Shadow DOM、CSS Modules、BEM 命名规范、动态加载/卸载样式表。

#### 路由冲突

主应用和子应用都有路由系统，需要约定路由命名空间：

```text
主应用路由：/app-a/...  → 挂载子应用 A
主应用路由：/app-b/...  → 挂载子应用 B
子应用内部路由：相对于各自的 base 路径
```

#### 应用间通信

```text
方案 1：基于 CustomEvent
  window.dispatchEvent(new CustomEvent('user-login', { detail: user }))

方案 2：发布订阅（EventBus）
  主应用暴露全局 eventBus，子应用 emit/on

方案 3：共享状态（props 传递 / qiankun 的 initGlobalState）
  适合父子应用传值

方案 4：URL / localStorage
  跨应用持久化数据
```

### 5. 性能考量

```text
问题：多个子应用都引入 React，导致重复加载

解决方案：
  1. Module Federation shared 字段 → 运行时只加载一份 React
  2. qiankun：子应用打包时 externals 排除 React，主应用提供
  3. 合理的按需加载：只有激活的子应用才加载其资源
```
