## Module Federation 是什么？

Webpack 5 引入的 **模块联邦（Module Federation）** 允许多个独立构建的应用在运行时互相暴露和消费模块，是目前最接近"真正代码共享"的微前端方案。

```text
传统方案（npm 包）：
  共享组件 → 发布 npm → 子应用安装 → 各自打包
  问题：每个子应用都打包一份，更新慢（要重新发布 npm）

Module Federation：
  Host 应用        Remote 应用
  ┌──────────┐    ┌──────────────────┐
  │ import   │    │ exposes:         │
  │  Button  │◄───│   './Button'     │
  │  from    │    │   './UserCard'   │
  │  remote  │    └──────────────────┘
  └──────────┘
  运行时动态加载，无需重新构建 Host
```

## 高频面试题

### 1. 核心概念：Host 与 Remote

```javascript
// remote 应用（被消费方）的 webpack.config.js
new ModuleFederationPlugin({
  name: 'remoteApp',           // 全局唯一名称
  filename: 'remoteEntry.js',  // 入口清单文件
  exposes: {
    // key: 对外暴露的路径名，value: 本地实际路径
    './Button': './src/components/Button',
    './utils': './src/utils/index',
  },
  shared: {
    react: { singleton: true, requiredVersion: '^18.0.0' },
    'react-dom': { singleton: true, requiredVersion: '^18.0.0' },
  },
});
```

```javascript
// host 应用（消费方）的 webpack.config.js
new ModuleFederationPlugin({
  name: 'hostApp',
  remotes: {
    // key: 引用别名，value: 远程应用名@入口地址
    remoteApp: 'remoteApp@http://localhost:3001/remoteEntry.js',
  },
  shared: {
    react: { singleton: true, requiredVersion: '^18.0.0' },
    'react-dom': { singleton: true, requiredVersion: '^18.0.0' },
  },
});
```

```javascript
// host 应用中消费远程模块（必须用 React.lazy + 动态 import）
const RemoteButton = React.lazy(() => import('remoteApp/Button'));

function App() {
  return (
    <Suspense fallback={<div>加载中...</div>}>
      <RemoteButton onClick={() => alert('来自远程应用！')} />
    </Suspense>
  );
}
```

### 2. 运行时加载原理

```text
用户访问 Host 应用
  │
  ├─ 加载 host 的 main.js
  │   └─ 遇到 import('remoteApp/Button')
  │
  ├─ 动态请求 http://localhost:3001/remoteEntry.js
  │   └─ 这是一个"清单"文件，记录了 remote 暴露了哪些模块、
  │      每个模块对应的 chunk 路径、以及依赖版本信息
  │
  ├─ 版本协商（shared 依赖）
  │   host 有 React 18.2 → remote 需要 ^18.0.0 → ✅ 复用 host 的 React
  │   不用再加载 remote 自己的 React（节省流量！）
  │
  └─ 加载 Button 对应的 chunk → 渲染组件
```

关键文件：`remoteEntry.js` 是联邦的核心，它暴露了一个全局对象（`window.remoteApp`），记录模块映射和版本信息。

### 3. shared 版本协商机制

这是面试中最常考的点：

```javascript
shared: {
  react: {
    singleton: true,        // 强制全局只有一个实例（React 必须 singleton）
    requiredVersion: '^18.0.0', // 声明需要的版本范围
    eager: false,           // 是否在入口处同步加载（false=懒加载，推荐）
    strictVersion: false,   // true=版本不匹配直接报错
  }
}
```

```text
版本协商规则（simplified）：

情况 1：Host React 18.2，Remote 需要 ^18.0.0
  → 满足，复用 Host 的 React ✅

情况 2：Host React 17.x，Remote 需要 ^18.0.0
  → 不满足，Remote 加载自己打包的 React 18 ⚠️（两份 React 并存）
  → 如果 singleton: true，会打印警告但强制复用 Host 的版本

情况 3：singleton: true + strictVersion: true + 版本不匹配
  → 直接抛错 ❌
```

> **面试高频**：为什么 React 必须设 `singleton: true`？
> 因为 React 使用 `React.__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED` 等全局状态，多个 React 实例会导致 Hooks 失效、Context 无法共享。

### 4. eager 异步加载陷阱

新手最常踩的坑：

```javascript
// ❌ 错误：直接在入口同步使用远程模块
import App from './App'; // App 内部用了 remoteApp/Button
ReactDOM.render(<App />, root);
// 报错：Shared module is not available for eager consumption

// ✅ 正确：通过 bootstrap 异步加载入口
// index.js（入口）
import('./bootstrap'); // 异步导入，给 MF 时间初始化

// bootstrap.js
import App from './App';
ReactDOM.render(<App />, root);
```

原因：Module Federation 的共享模块初始化是异步的，入口文件必须给 MF 一个"时间窗口"完成协商。

### 5. Vite 中的 Module Federation

Vite 原生不支持 MF，需要 `@originjs/vite-plugin-federation`：

```javascript
// vite.config.ts（remote 应用）
import federation from '@originjs/vite-plugin-federation';

export default defineConfig({
  plugins: [
    federation({
      name: 'remoteApp',
      filename: 'remoteEntry.js',
      exposes: {
        './Button': './src/components/Button.vue',
      },
      shared: ['vue'],
    }),
  ],
  build: {
    target: 'esnext',  // MF 依赖动态 import，必须设置
    minify: false,     // 建议关闭，方便调试
  },
});
```

```javascript
// vite.config.ts（host 应用）
import federation from '@originjs/vite-plugin-federation';

export default defineConfig({
  plugins: [
    federation({
      name: 'hostApp',
      remotes: {
        remoteApp: 'http://localhost:5001/assets/remoteEntry.js',
      },
      shared: ['vue'],
    }),
  ],
  build: { target: 'esnext' },
});
```

**Vite MF 与 Webpack MF 的差异：**

```text
| 特性           | Webpack MF            | Vite MF（插件）        |
|----------------|----------------------|----------------------|
| 开发环境支持   | ✅ 完整              | ⚠️ 部分（需 build）   |
| 生态成熟度     | ✅ 成熟              | ⚠️ 仍在演进           |
| 构建产物格式   | UMD/CJS              | ESM                  |
| 动态 remote    | ✅ 支持              | ⚠️ 需额外处理         |
```

### 6. 动态 Remote URL（运行时配置）

生产环境中 Remote 的地址通常来自配置中心，不能硬编码：

```javascript
// 动态配置远程地址
async function loadRemoteModule(url, scope, module) {
  // 动态插入 script 标签
  await new Promise((resolve, reject) => {
    const script = document.createElement('script');
    script.src = url;
    script.onload = resolve;
    script.onerror = reject;
    document.head.appendChild(script);
  });

  // 初始化共享作用域
  await __webpack_init_sharing__('default');
  const container = window[scope];
  await container.init(__webpack_share_scopes__.default);

  // 获取模块
  const factory = await container.get(module);
  return factory();
}

// 使用
const { default: Button } = await loadRemoteModule(
  'https://cdn.example.com/remoteEntry.js',
  'remoteApp',
  './Button'
);
```

### 7. 常见面试追问

**Q：Module Federation 和 npm 包有什么本质区别？**

```text
npm 包：
  - 构建时确定，打进 bundle
  - 更新需要重新发布 + 重新安装 + 重新构建消费方
  - 适合稳定的基础库

Module Federation：
  - 运行时加载，无需重新构建消费方
  - Remote 发布新版本，Host 刷新页面即可获取
  - 适合频繁迭代的业务组件
```

**Q：使用 MF 后 Remote 应用挂了怎么办？**

```javascript
// 使用 ErrorBoundary + Suspense 优雅降级
const RemoteWidget = React.lazy(() =>
  import('remoteApp/Widget').catch(() => ({
    default: () => <div>组件加载失败，请刷新重试</div>
  }))
);
```
