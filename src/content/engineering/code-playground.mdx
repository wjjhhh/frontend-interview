## 面试题：如何实现一个在线代码执行器？

这是一道综合面试题，考查对**浏览器安全模型、动态代码执行、编译原理**的理解。

系统需要满足：
- 用户在编辑器输入 JSX 代码
- 浏览器端实时编译 + 渲染预览
- 隔离执行环境（不能访问主页面数据）
- 捕获运行时错误并展示

## 整体架构

```text
┌─────────────────────────────────────────────────────────────┐
│  主页面（Next.js / React App）                               │
│                                                             │
│  ┌───────────────┐         ┌─────────────────────────────┐ │
│  │  代码编辑器    │         │      iframe（隔离沙箱）       │ │
│  │  <textarea>   │──构建──▶│                             │ │
│  │  或 Monaco    │  srcdoc │  React + ReactDOM（CDN）     │ │
│  └───────────────┘         │  @babel/standalone（CDN）   │ │
│                            │                             │ │
│  ① 用户输入 JSX            │  ② Babel 编译 JSX → JS     │ │
│  ③ 防抖 600ms              │  ④ ReactDOM.createRoot 渲染 │ │
│  ⑤ 更新 srcdoc             │  ⑥ 错误上报 window.onerror │ │
│                            └─────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

## 高频面试题

### 1. 为什么用 iframe 而不是直接 eval()？

```javascript
// ❌ 方案1：直接 eval（危险）
eval(userCode);
// 问题：
// - 与主页面共享同一个 JS 执行上下文
// - 用户代码可以访问/修改主页面的所有变量
// - 无法隔离 localStorage、cookie、DOM 操作
// - eval 的代码无法被 GC（内存泄漏）

// ❌ 方案2：new Function（略好，但仍不安全）
const fn = new Function('React', 'ReactDOM', userCode);
fn(React, ReactDOM);
// 仍然共享主页面 window，可以访问 document、fetch 等

// ✅ 方案3：sandbox iframe（正确）
// iframe 有独立的：
// - window 对象
// - document / DOM 树
// - localStorage（同源时独立作用域）
// - JS 执行上下文（主页面变量完全不可访问）
// - 崩溃不影响主页面（子进程隔离）
```

### 2. srcdoc vs src：如何将代码注入 iframe？

```html
<!-- ❌ src 方案：需要服务器端接口 -->
<iframe src="/api/preview?code=xxx"></iframe>
<!-- 需要把代码发送给服务器，服务器返回 HTML，延迟高 -->

<!-- ✅ srcdoc 方案：纯客户端，无需服务器 -->
<iframe srcdoc="<!DOCTYPE html><html>...</html>"></iframe>
<!-- 直接将 HTML 字符串设为 iframe 内容，零延迟 -->
```

```typescript
// 构建 iframe 的完整 HTML
function buildIframeSrc(userCode: string): string {
  return `<!DOCTYPE html>
<html>
<head>
  <meta charset="UTF-8">
  <!-- 在 iframe 内部加载 React + Babel，与主页面的 React 完全隔离 -->
  <script src="https://unpkg.com/react@18/umd/react.development.js"></script>
  <script src="https://unpkg.com/react-dom@18/umd/react-dom.development.js"></script>
  <script src="https://unpkg.com/@babel/standalone/babel.min.js"></script>
</head>
<body>
  <div id="root"></div>
  <script type="text/babel" data-presets="react">
    // 解构常用 Hooks，让用户代码无需 import
    const { useState, useEffect, useRef, useMemo,
            useCallback, useReducer } = React;

    ${userCode}

    // 约定：用户需要定义 App 组件作为入口
    ReactDOM.createRoot(document.getElementById('root'))
      .render(React.createElement(App));
  </script>
</body>
</html>`;
}
```

### 3. iframe sandbox 属性：最小权限原则

```html
<!-- sandbox 属性默认禁止一切，按需开放 -->
<iframe
  srcdoc="..."
  sandbox="allow-scripts"
  title="Code Preview"
/>

<!-- sandbox 各权限含义 -->
<!--
  allow-scripts       ← 允许执行 JS（必须开启，否则代码无法运行）
  allow-same-origin   ← 允许访问同源资源（不开启则 localStorage 被禁）
  allow-forms         ← 允许表单提交
  allow-popups        ← 允许弹窗
  allow-top-navigation ← 允许修改父页面 URL（危险！不开启）
-->

<!-- ⚠️ 注意：同时设置 allow-scripts + allow-same-origin 会削弱隔离 -->
<!-- 因为脚本可以通过 parent.xxx 访问父页面 -->
<!-- 纯代码演示场景只需 allow-scripts 即可 -->
```

### 4. @babel/standalone：浏览器端 JSX 编译原理

```html
<!-- type="text/babel" 告诉 Babel standalone 拦截并编译这段脚本 -->
<script type="text/babel" data-presets="react">
  const App = () => <h1>Hello</h1>;
  // ↓ Babel 在运行时将其转换为：
  // const App = () => React.createElement("h1", null, "Hello");
  ReactDOM.createRoot(root).render(React.createElement(App));
</script>
```

```javascript
// 或者用 JS API 手动编译（更灵活）
const result = Babel.transform(userCode, {
  presets: ['react'],           // 编译 JSX
  plugins: ['transform-modules-commonjs'], // 可选：CommonJS 模块
  filename: 'playground.jsx',
});

const compiledCode = result.code;
// 再通过 new Function 执行，或注入到 script 标签
```

**为什么不在主页面编译再传给 iframe？**

```text
方案A（主页面编译）：
  主页面安装 @babel/core → 编译 JSX → 发 postMessage → iframe 执行
  优点：主页面可用更强大的 Babel 配置
  缺点：@babel/core 打包体积大（~1MB），需要 postMessage 通信

方案B（iframe 内编译，本文采用）：
  iframe 从 CDN 加载 @babel/standalone → 自行编译执行
  优点：主页面零体积，架构简单
  缺点：依赖 CDN，首次加载慢（可预加载缓解）
```

### 5. 错误处理：三层防护

```typescript
function buildIframeSrc(code: string): string {
  return `...
  <div id="error" style="display:none; color:red; padding:12px; font-family:monospace; white-space:pre-wrap;"></div>

  <script>
    // 第一层：全局未捕获错误（运行时 JS 错误）
    window.addEventListener('error', function(e) {
      var el = document.getElementById('error');
      el.style.display = 'block';
      el.textContent = e.error ? e.error.stack : e.message;
    });

    // 第二层：未处理的 Promise rejection
    window.addEventListener('unhandledrejection', function(e) {
      var el = document.getElementById('error');
      el.style.display = 'block';
      el.textContent = 'Unhandled Promise: ' + e.reason;
    });
  </script>

  <script type="text/babel" data-presets="react">
    // 第三层：try/catch 包裹渲染逻辑
    try {
      ${code}
      ReactDOM.createRoot(document.getElementById('root'))
        .render(React.createElement(App));
    } catch(e) {
      document.getElementById('error').style.display = 'block';
      document.getElementById('error').textContent = e.toString();
    }
  </script>`;
}
```

### 6. 实时预览：防抖 + 避免频繁重建 iframe

```typescript
// ❌ 每次 onChange 都更新 srcdoc：输入中途频繁重建 iframe，体验差
function Editor() {
  const [code, setCode] = useState('');
  const [src, setSrc] = useState('');

  return (
    <textarea
      onChange={(e) => {
        setCode(e.target.value);
        setSrc(buildIframeSrc(e.target.value)); // 每次按键都重建！
      }}
    />
  );
}

// ✅ 防抖：停止输入 600ms 后再更新
function Editor() {
  const [code, setCode] = useState(defaultCode);
  const [iframeSrc, setIframeSrc] = useState(() => buildIframeSrc(defaultCode));
  const timerRef = useRef<ReturnType<typeof setTimeout>>(undefined);

  const handleChange = useCallback((newCode: string) => {
    setCode(newCode);
    clearTimeout(timerRef.current);
    timerRef.current = setTimeout(() => {
      setIframeSrc(buildIframeSrc(newCode)); // 停止输入后才重建
    }, 600);
  }, []);

  useEffect(() => () => clearTimeout(timerRef.current), []); // 组件卸载清理

  return (
    <>
      <textarea value={code} onChange={(e) => handleChange(e.target.value)} />
      <iframe srcDoc={iframeSrc} sandbox="allow-scripts" />
    </>
  );
}
```

### 7. 字符串注入安全：转义用户代码

```typescript
// 问题：用户代码中含有反引号 ` 或 ${} 会破坏模板字符串
const userCode = 'const s = `hello ${name}`'; // 破坏外层模板！

// ❌ 直接拼接（有注入风险）
return `<script type="text/babel">${userCode}</script>`;

// ✅ 方案1：转义特殊字符（模板字符串内）
function buildIframeSrc(code: string): string {
  const safe = code
    .replace(/\\/g, '\\\\')   // 反斜杠先转义
    .replace(/`/g, '\\`')     // 反引号
    .replace(/\$/g, '\\$');   // $（防止 ${} 插值）

  return `...
  <script type="text/babel">
    ${safe}
  </script>`;
}

// ✅ 方案2：用 JSON.stringify 序列化，再 eval（更安全）
function buildIframeSrc(code: string): string {
  const serialized = JSON.stringify(code); // 自动处理所有特殊字符

  return `...
  <script>
    // 通过 JSON 序列化避免注入
    const code = ${serialized};
    Babel.transform(code, { presets: ['react'] });
    // 再执行编译结果...
  </script>`;
}
```

### 8. 终极优化：iframe 静态壳 + postMessage

上面 srcdoc 方案的致命问题：**每次代码变动都重建 iframe，导致 React、Babel 等 CDN 资源反复加载**。

```text
srcdoc 方案（每次重建）：
  用户输入 → 防抖 600ms → 重建整个 iframe HTML → 浏览器解析
  → 重新加载 React CDN ← 命中缓存但仍需解析执行 ~100ms
  → 重新加载 Babel CDN ← 命中缓存但仍需解析执行 ~200ms
  → 编译用户代码 → 渲染

postMessage 方案（只加载一次）：
  iframe 初始化时加载一次 React + Babel（静态壳）
  → 用户输入 → 防抖 600ms → postMessage 传代码
  → iframe 内部编译 + 渲染 ← 无需重新加载任何资源，~10ms
```

**核心架构：**

```text
┌──────────────────────────────────────────────────────────────┐
│  主页面                                                       │
│                                                              │
│  ┌───────────────┐    postMessage({ type:'run', code })      │
│  │  代码编辑器    │  ─────────────────────────────────────▶   │
│  └───────────────┘                                           │
│                       postMessage({ type:'iframe-ready' })   │
│  ┌───────────────┐  ◀─────────────────────────────────────   │
│  │  iframe 预览   │    iframe 静态壳（只加载一次，常驻内存）    │
│  │  sandbox=      │    · React + ReactDOM（CDN，一次性加载）  │
│  │  allow-scripts │    · @babel/standalone（CDN，一次性加载） │
│  └───────────────┘    · runCode() 接收代码，编译执行          │
└──────────────────────────────────────────────────────────────┘
```

**Step 1：iframe 静态壳（只初始化一次）**

```html
<!-- 这个 HTML 通过 srcdoc 注入一次，之后永不重建 -->
<!DOCTYPE html>
<html>
<head>
  <script src="https://unpkg.com/react@18/umd/react.development.js"></script>
  <script src="https://unpkg.com/react-dom@18/umd/react-dom.development.js"></script>
  <script src="https://unpkg.com/@babel/standalone/babel.min.js"></script>
</head>
<body>
  <div id="root"></div>
  <div id="error" style="display:none"></div>
  <script>
    var reactRoot = null;
    // 预解构 Hooks，用户代码无需 import
    var HOOKS = ['useState','useEffect','useRef','useMemo',
                 'useCallback','useReducer','useContext'];

    function runCode(code) {
      // 清除旧错误
      document.getElementById('error').style.display = 'none';

      // 卸载旧组件（避免内存泄漏和状态残留）
      if (reactRoot) {
        try { reactRoot.unmount(); } catch(e) {}
        reactRoot = null;
      }
      document.getElementById('root').innerHTML = '';

      try {
        // Babel 编译 JSX → JS（Babel 已在内存中，无需重新加载）
        var transformed = Babel.transform(code, { presets: ['react'] }).code;

        // 构建沙箱作用域
        var scope = { React: React, ReactDOM: ReactDOM };
        HOOKS.forEach(function(h) { if (React[h]) scope[h] = React[h]; });

        // 执行编译后的代码，获取 App 组件
        var keys = Object.keys(scope);
        var vals = keys.map(function(k) { return scope[k]; });
        var fn = new Function(keys.join(','),
          transformed + '\nreturn typeof App !== "undefined" ? App : null;');
        var App = fn.apply(null, vals);

        // 渲染
        if (App) {
          reactRoot = ReactDOM.createRoot(document.getElementById('root'));
          reactRoot.render(React.createElement(App));
        }
      } catch(e) {
        document.getElementById('error').style.display = 'block';
        document.getElementById('error').textContent = e.toString();
      }
    }

    // 监听主页面发来的代码
    window.addEventListener('message', function(e) {
      if (e.data && e.data.type === 'run') runCode(e.data.code);
    });

    // 通知主页面：iframe 已就绪（CDN 资源加载完成）
    window.parent.postMessage({ type: 'iframe-ready' }, '*');
  </script>
</body>
</html>
```

**Step 2：主页面通信逻辑**

```tsx
function ReactPlayground() {
  const [code, setCode] = useState(defaultCode);
  const iframeRef = useRef<HTMLIFrameElement>(null);
  const iframeReadyRef = useRef(false);
  const codeRef = useRef(code);
  const debounceRef = useRef<ReturnType<typeof setTimeout>>();

  // 保持 ref 同步（闭包陷阱防护）
  useEffect(() => { codeRef.current = code; }, [code]);

  // 监听 iframe 就绪信号
  useEffect(() => {
    function handleMessage(e: MessageEvent) {
      if (e.data?.type === 'iframe-ready') {
        iframeReadyRef.current = true;
        // 就绪后立即发送当前代码（首次渲染）
        iframeRef.current?.contentWindow?.postMessage(
          { type: 'run', code: codeRef.current }, '*'
        );
      }
    }
    window.addEventListener('message', handleMessage);
    return () => window.removeEventListener('message', handleMessage);
  }, []);

  // 代码变动后 600ms 通过 postMessage 更新（不重建 iframe）
  useEffect(() => {
    clearTimeout(debounceRef.current);
    debounceRef.current = setTimeout(() => {
      if (iframeReadyRef.current) {
        iframeRef.current?.contentWindow?.postMessage(
          { type: 'run', code }, '*'
        );
      }
    }, 600);
    return () => clearTimeout(debounceRef.current);
  }, [code]);

  return (
    <>
      <textarea value={code} onChange={(e) => setCode(e.target.value)} />
      {/* srcdoc 只设置一次，之后 iframe 常驻内存 */}
      <iframe ref={iframeRef} srcDoc={IFRAME_SHELL} sandbox="allow-scripts" />
    </>
  );
}
```

**关键设计点：**

```text
1. iframe-ready 握手
   iframe 加载完 CDN 后主动通知主页面 → 主页面才开始发送代码
   避免 postMessage 发早了、iframe 还没准备好的时序问题

2. 旧组件卸载
   每次 runCode 前调用 reactRoot.unmount()
   防止多次渲染导致内存泄漏和状态残留

3. new Function 沙箱
   用 new Function 而非 eval，显式传入作用域变量
   用户代码只能访问注入的 React/Hooks，不能访问 iframe 的全局变量

4. ref 同步
   codeRef.current 始终指向最新代码
   解决 iframe-ready 回调中闭包捕获旧值的问题

5. 性能对比
   srcdoc 重建：每次 ~300ms（CDN 解析 + Babel 初始化 + 编译 + 渲染）
   postMessage：每次 ~10ms（仅编译 + 渲染）
   提升约 30 倍
```

### 如何支持多文件（App.jsx + utils.js）？

### 生产级 Playground 的优化方向

```text
1. iframe 静态壳 + postMessage（上文第 8 节已实现）
   一次性加载 CDN 资源，后续代码更新只通过消息传递
   性能提升 ~30 倍，是最关键的架构优化

2. 编辑器升级
   textarea → Monaco Editor（VS Code 同款，有语法高亮、自动补全）

3. CDN 资源预加载

4. 编译移至 Web Worker
   将 Babel.transform 放到 Worker 线程
   避免大型代码编译时阻塞主线程 UI

5. TypeScript 支持
   @babel/standalone 支持 data-presets="typescript,react"
   或使用 @typescript/vfs + TypeScript compiler API

6. 错误提示增强
   解析 Babel 编译错误的 loc（行列号）
   在编辑器对应位置显示红色波浪线
```

### esm.sh 方案：运行时按需加载 npm 依赖

上述方案用 unpkg 加载的是 **UMD 格式**（挂载到 `window.React`），而 esm.sh 提供另一种思路：把 npm 包实时转换为 **ES Module 格式**，让浏览器原生 `import`。

**为什么浏览器能直接 import esm.sh 的 URL？**

```html
<script type="module">
  // 浏览器看到完整 URL，直接发 HTTP 请求取回模块
  import React from 'https://esm.sh/react@18'
  import { useState } from 'https://esm.sh/react@18'
</script>
```

浏览器原生支持 `type="module"` + URL import，不需要 bundler，不需要 node_modules。esm.sh 在服务端把 npm 的 CJS 格式实时转换成 ESM：

```js
// npm 原始包（CJS）
const React = require('./react')
module.exports = React

// esm.sh 返回的（ESM）
export default React
export { useState, useEffect, ... }
```

**加载链路：**

```text
import 'https://esm.sh/react'
        ↓
  浏览器发 GET 请求
        ↓
  esm.sh 服务器检查缓存
  → 没有：拉 npm 包 → 转成 ESM → 返回
  → 有：直接返回缓存
        ↓
  浏览器解析 JS，遇到子依赖继续发请求（瀑布式）
        ↓
  所有模块就绪 → 代码执行
```

**在 Playground 中的用法：**

```js
// 把用户代码里的裸模块名替换为 esm.sh URL
// import React from 'react'  →  import React from 'https://esm.sh/react'

const transformed = userCode.replace(
  /from ['"]([^./][^'"]*)['"]/g,
  (_, pkg) => `from 'https://esm.sh/${pkg}'`
)
```

配合 Import Map 更优雅：

```html
<script type="importmap">
{
  "imports": {
    "react": "https://esm.sh/react@18",
    "react-dom": "https://esm.sh/react-dom@18"
  }
}
</script>
<script type="module">
  import React from 'react'  // 自动映射到 esm.sh
</script>
```

**网络慢会怎样？**

```text
问题1：模块加载超时，import 语句挂起，预览区白屏
问题2：依赖链长时（A 依赖 B 依赖 C），每个都要单独请求，瀑布式延迟叠加
问题3：esm.sh 是动态转换服务，响应时间不稳定，不像静态 CDN 有缓存保证
问题4：超时后直接抛错（Failed to fetch dynamically imported module），无自动重试
```

应对方案：
- 换更快的 CDN（jsDelivr: `cdn.jsdelivr.net/npm/react/+esm`）
- Service Worker 缓存 esm.sh 响应，首次慢但之后秒开
- 切换为 esbuild-wasm 本地打包，彻底摆脱运行时网络依赖

**浏览器与 Node.js 兼容性：**

```text
浏览器支持（原生 ESM URL import）：
  Chrome 61+（2017）  ✅
  Firefox 60+（2018）  ✅
  Safari 10.1+（2017）  ✅
  Edge 16+（2017）  ✅
  IE  ✗  完全不支持（已死）
  → 现在基本无限制

Node.js：
  无法直接 import URL，这是浏览器独有能力
  import React from 'https://esm.sh/react'  // Node.js ❌ 报错
  // Error: Only file: and data: URLs are supported
  → esm.sh 方案是纯浏览器方案，与 Node 版本无关
```

**与 UMD 方案对比：**

```text
UMD（unpkg）：
  <script src="unpkg.com/react"></script>
  → 挂载到 window.React 全局变量
  → 所有依赖一次性加载，简单可控

esm.sh（ESM）：
  import React from 'https://esm.sh/react'
  → 通过模块系统加载，天然 tree-shaking
  → 支持按需加载任意 npm 包，适合多依赖场景
  → 但依赖运行时网络，延迟不可控
```

### CodeSandbox / StackBlitz 的底层方案对比

```text
本文方案（轻量）：
  iframe srcdoc + CDN Babel
  适合：简单组件预览、教学演示
  限制：单文件、无 npm 依赖

CodeSandbox Sandpack：
  Service Worker 虚拟文件系统 + 内置 bundler
  适合：完整项目预览，支持 npm 包
  原理：worker 内运行 Parcel bundler

StackBlitz WebContainers：
  WebAssembly 版 Node.js 运行在浏览器中
  适合：完整 Node.js 项目，可运行 vite dev server
  原理：WASM + SharedArrayBuffer + Service Worker
  限制：需要 COOP/COEP 响应头（跨域隔离）
```
